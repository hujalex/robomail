❯ where do we get the NPM_TOKEN                                                 

⏺ You create it in your npm account:

  1. Go to https://npmjs.com → log in → click your avatar → Access Tokens
  2. Generate New Token → choose Granular Access Token (recommended) or
  Automation (for CI)
    - Granular: lets you scope it to specific packages/orgs
    - Automation: bypasses 2FA, good for CI pipelines
  3. Copy the token (shown once)

  Then set it wherever you run fern generate --group publish:

  Locally:
  NPM_TOKEN=npm_xxxx fern generate --group publish

  In CI (e.g. GitHub Actions):
  - Add it as a repository secret: Settings → Secrets → NPM_TOKEN
  - Reference it in your workflow: ${{ secrets.NPM_TOKEN }}

  Note: the robomail package name needs to be available on npm, or you need to
  be logged into an account that owns it. If someone already has robomail, you'd
   use a scoped name like @yourorg/robomail — just update package-name in both
  places in generators.yml.

❯ is an automation token just a granular without 2fa enabled?                   

⏺ Not exactly. There are three token types on npm:

  - Classic (legacy) — full account access, being phased out
  - Granular — scoped to specific packages, orgs, IP ranges, and expiry dates.
  Respects your 2FA setting
  - Automation — same full access as a classic token, but explicitly bypasses
  2FA for CI use. No scoping options

  So Automation is more permissive than Granular (broader access, no 2FA), while
   Granular is more secure but requires you to configure the scope manually.

  For publishing a single package from CI, Granular is the better choice — scope
   it to just robomail, set an expiry, and you limit the blast radius if the
  token leaks.

❯ should we make another repository seperate from robomail for holding the sdk  

⏺ Yes, that's the standard approach. A separate repo lets you:

  - Version and release the SDK independently from the API server
  - Keep the SDK's issues, PRs, and changelog focused on the client surface
  - Let consumers file bugs without access to server internals

  The typical setup is:

  - robomail (this repo) — API server, generates the SDK via fern generate 
  --group publish in CI when you cut a release
  - robomail-node (new repo) — the generated SDK lives here, published to npm as
   robomail

  Fern supports this natively — in generators.yml you can point the publish
  group at a GitHub repo instead of (or in addition to) npm:

  publish:
    generators:
      - name: fern-typescript-sdk
        version: 3.70.7
        github:
          repository: your-org/robomail-node
        output:
          location: npm
          package-name: robomail
          token: ${NPM_TOKEN}

  Fern will open a PR against robomail-node with the regenerated SDK on each
  run, which gives you a review step before publishing. That's the cleanest
  workflow if you want auditability.

❯ To set up **Trusted Publishing** (OIDC authentication) for your packages, you 
   need to configure it at the **individual package level** rather than in your 
   global account settings.                                                     
                                                                                
  Here is exactly where to head and how to configure it:                        
                                                                                
  ### 1. Head to Your Package Settings on npm                                   
                                                                                
  1. Go to **npmjs.com** and make sure you are logged into your account         
  (`hujalex`).                                                                  
  2. Navigate directly to your specific package's access settings by replacing  
  your package name in the URL:                                                 
  `https://www.npmjs.com/package/<YOUR_PACKAGE_NAME>/access`                    
  3. Scroll down to the **Trusted Publisher** section.                          
  4. Under **Select your publisher**, click on your CI/CD provider (e.g.,       
  **GitHub Actions**, **GitLab CI/CD**, or **CircleCI**).                       
                                                                                
  ### 2. Fill Out the Publisher Configuration                                   
                                                                                
  For **GitHub Actions**, you will need to fill out the following fields:       
                                                                                
  * **Organization or user:** `hujalex` (or your specific GitHub organization   
  name).                                                                        
  * **Repository:** The name of your GitHub repository.                         
  * **Workflow filename:** The exact name of your release workflow file (e.g.,  
  `release.yml` or `publish.yml`). *Note: This file must exist inside           
  `.github/workflows/` in your repository.*                                     
                                                                                
  ### 3. Update Your CI/CD Workflow File                                        
                                                                                
  Your automation needs permission to mint the OIDC token. In your GitHub       
  Actions workflow file, you must explicitly declare the `id-token: write`      
  permission.                                                                   
                                                                                
  Here is a streamlined example:                                                
                                                                                
  ```yaml                                                                       
  name: Publish Package                                                         
  on:                                                                           
    push:                                                                       
      tags:                                                                     
        - 'v*'                                                                  
                                                                                
  permissions:                                                                  
    id-token: write    # 👈 Crucial for OIDC Trusted Publishing                 
    contents: read                                                              
                                                                                
  jobs:                                                                         
    publish:                                                                    
      runs-on: ubuntu-latest                                                    
      steps:                                                                    
        - uses: actions/checkout@v4                                             
        - uses: actions/setup-node@v4                                           
          with:                                                                 
            node-version: '24' # Node 24+ natively ships with a compatible npm  
  version                                                                       
            registry-url: 'https://registry.npmjs.org'                          
                                                                                
        - run: npm ci                                                           
        - run: npm run build --if-present                                       
        - run: npm publish --provenance # Generates cryptographic proof of the  
  build                                                                         
                                                                                
  ```                                                                           
                                                                                
  > ⚠️ **Important Version Requirement:** Trusted publishing requires **npm CLI 
   version 11.5.1 or later**. Using Node 24+ fulfills this out of the box. If   
  your workflow uses an older Node version (like Node 20), you must add an      
  explicit upgrade step (`- run: npm install -g npm@latest`) before running the 
   publish command.                                                             
                                                                                
  For deeper details on restricting traditional long-lived tokens and           
  tightening down your package security, check out the official npm guide on    
  [Trusted publishing for npm                                                   
  packages](https://docs.npmjs.com/trusted-publishers/).
